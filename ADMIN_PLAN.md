# Admin Module — Plan

> Status: **planned, with a UI prototype shipped** (`/admin`, localStorage-only).
> Backend wiring is staged below — review before we connect anything.
>
> This is the document we agree on first. Once approved, the work below is
> mechanical.

The Admin module is the spine of the multi-tenant story. Every other tool
(APIStress + PostWomen) sits on top of it: who can log in, what they see, and
what data is theirs. So we settle the design here first.

---

## 1. The user story

> *"As an admin, I create a team. The system gives me an auto-generated key.
>  I share that key with the team's members. When they paste it on the login
>  screen, they're inside the tool — and what they see is **scoped to that
>  team's runs and collections only**. The team name is shown at the top
>  everywhere they go. They can never see another team's data."*

That's the whole thing. Every decision below serves this story.

---

## 2. The data model

### New tables

```sql
CREATE TABLE teams (
  id           UUID PRIMARY KEY,
  name         TEXT NOT NULL UNIQUE,
  description  TEXT NOT NULL DEFAULT '',
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  created_by   TEXT NOT NULL DEFAULT 'admin',
  is_active    BOOL NOT NULL DEFAULT TRUE
);

CREATE TABLE team_keys (
  id            UUID PRIMARY KEY,
  team_id       UUID NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
  key_hash      BYTEA NOT NULL,             -- bcrypt of the actual key
  key_prefix    TEXT NOT NULL,              -- first 8 chars, for "Last used: prod-xxxx"
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_used_at  TIMESTAMPTZ,
  revoked_at    TIMESTAMPTZ                 -- soft-delete on rotation
);
CREATE INDEX idx_team_keys_team_active ON team_keys(team_id) WHERE revoked_at IS NULL;

-- Optional: track members per team so admins can see who's in.
CREATE TABLE team_members (
  team_id       UUID NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
  display_name  TEXT NOT NULL,              -- whatever they typed in "Your name"
  first_seen_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_seen_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  request_count INT NOT NULL DEFAULT 0,
  PRIMARY KEY (team_id, display_name)
);
```

### Modified existing tables

Every table that holds user-generated data gets a `team_id`:

```sql
ALTER TABLE runs            ADD COLUMN team_id UUID REFERENCES teams(id);
ALTER TABLE tests           ADD COLUMN team_id UUID REFERENCES teams(id);
ALTER TABLE environments    ADD COLUMN team_id UUID REFERENCES teams(id);
ALTER TABLE pw_workspaces   ADD COLUMN team_id UUID REFERENCES teams(id);
ALTER TABLE pw_history      ADD COLUMN team_id UUID REFERENCES teams(id);
```

Existing rows from before the migration get assigned to a special **"Legacy"
team** (auto-created during migration). Admins can re-assign or delete.

### Why hash the key?

Plain `key` columns leak if the DB is dumped. We **store bcrypt(key)** and a
short prefix (`prod-a3f2`) for display. Validation hashes the incoming key and
looks it up.

## 3. Authentication flow

### Today
```
User → enters CH_ACCESS_KEY (env var) → matches → in.
```

### After Admin module ships
```
User → enters team key (generated by admin) → backend hashes & looks up →
        finds team_keys row → not revoked → marks last_used_at →
        issues a session cookie carrying { team_id, key_id } →
        every subsequent request is scoped to that team_id.
```

The user **never knows** the team_id. They just paste a key. The backend does
the lookup. No change for them — except their data is now isolated.

### Backwards-compat with existing `CH_ACCESS_KEY`

For 30 days during the rollout: the env-var key still works and acts like the
"Legacy" team. Then we delete it.

## 4. Admin authentication

The `/admin` route needs its own gate, separate from the main app key.

Two simple options:

| Option | Pros | Cons |
|---|---|---|
| **Env var `CH_ADMIN_KEY`** (separate from team keys) | Trivial. Works today. | One shared admin key = same problem at scale. |
| **`is_admin` flag on a special team** | Reuses team-key infra. | Slight chicken-and-egg on bootstrap. |

**Verdict: env-var `CH_ADMIN_KEY`** for v1. We're talking ~3 admins, rotated
manually. Phase-2 multi-tenant (NEXTPLAN.md) replaces it with proper RBAC.

The admin passphrase is required:
1. To open `/admin` at all (gate on the page).
2. As a header (`X-Admin-Key`) on every `/api/admin/*` endpoint.

## 5. API surface

All `/api/admin/*` routes require `X-Admin-Key`.

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/admin/auth`              | Validate the admin passphrase (returns 200 / 401). UI uses this on the login screen. |
| `GET`  | `/api/admin/teams`             | List teams + member counts + last-key-used timestamp. |
| `POST` | `/api/admin/teams`             | Create a team. Body: `{name, description}`. Returns `{team, plain_key}` — the plaintext key is shown **once**, never again. |
| `PATCH`| `/api/admin/teams/:id`         | Rename / re-describe / activate / deactivate. |
| `DELETE`| `/api/admin/teams/:id`        | Delete (cascades all keys + scoped data, with a confirm). |
| `POST` | `/api/admin/teams/:id/rotate-key` | Generate a new key for the team, revoke the old one. Returns the new plaintext once. |
| `GET`  | `/api/admin/teams/:id/members` | List `team_members` for a team. |
| `GET`  | `/api/admin/audit?limit=…`     | Audit feed (key generated, key rotated, team deleted…). |
| `GET`  | `/api/admin/stats`             | Tile counts: # teams, # runs by team, # active members in last 7 days. |

User-side stays untouched **except**:

- `POST /api/auth/login` now accepts a *team key* (and falls back to legacy
  `CH_ACCESS_KEY` during rollout).
- Login response body includes `{team_id, team_name}` so the UI can show it.
- `/api/runs`, `/api/postwomen/*` etc. all gain an implicit `WHERE team_id = $session.team_id`.

## 6. Admin UI walkthrough (already prototyped)

The route `/admin` is **separate** from the main shell — no sidebar, no other
tools visible. Pure admin chrome.

### 6a. Login screen
- Centered card with the Choice Techlab mark
- "Admin passphrase" field (password input)
- Submit → `POST /api/admin/auth` → on success store the result in
  `sessionStorage` (so closing the tab signs out). On fail, toast.

### 6b. Console
- **Top bar:** Back to home · Choice Techlab mark · "Admin console" title ·
  Sign-out admin button.
- **Notice banner (orange):** explains current scope (we'll remove once
  backend lands).
- **3 KPI tiles:** Teams · Active keys · Total members.
- **Search + "+ New team"** action row.
- **Team list — one card per team:**
  - Avatar (first 2 letters of team name on a brand gradient)
  - Name + description
  - Member chip + created-on chip
  - Rotate / Delete buttons
  - **Key block** — copy-to-clipboard mono row with "Last used: 2 hours ago"
- **Create-team modal:** name + description → key auto-generates with the
  pattern `<slug>-<random>-<random>-<random>` (e.g. `backend-a3f2-9z1k-77pq`).
- **Future tabs (not in v1):** Audit log · Members · Quotas (mirror NEXTPLAN.md
  Phase 4).

### 6c. Behaviour the user sees inside the team

- Login page accepts the team key as today.
- After login, the **sidebar bottom (APIStress) and top bar (PostWomen)** show
  a small chip with the team name (e.g. `BACKEND`).
- All History / Reports / Collections lists are scoped — users genuinely
  cannot see other teams' data.
- A user typing their name in "Your name" creates a `team_members` row for
  visibility in the admin console; nothing more.

## 7. Scoping every existing query

This is the part the multi-tab / report / cost work all silently depends on.
The pattern is:

```go
// auth/middleware.go (new)
func TeamAuth(c *gin.Context) {
    key := extractKey(c) // X-Access-Key | ?key= | Bearer
    teamID, err := lookup(key)
    if err != nil { c.AbortWithStatus(401); return }
    c.Set("team_id", teamID)
    c.Next()
}

// helpers/scope.go (new)
func TeamID(c *gin.Context) string { return c.GetString("team_id") }
```

Every existing handler gets ONE change at the top:

```go
team := helpers.TeamID(c)
// reads:    WHERE team_id = $team
// writes:   INSERT … team_id = $team
```

Done in one mechanical pass across the existing handlers. Roughly 20 query
sites.

## 8. Edge cases we handle deliberately

| Case | Behaviour |
|---|---|
| **Admin deletes a team mid-run** | Active runs in that team are sent a context cancel; UI shows "team deleted, contact admin". |
| **Admin rotates a key** | All sessions using the old key get 401 on next request. UI bounces them to the login page with a "Your key was rotated by an admin." toast. |
| **User shares a report URL across teams** | Reports endpoint now joins `team_id` — cross-team URL access returns 404, not 403 (no info leak). |
| **PostWomen "Send to APIStress" across team scope** | The pre-fill query string carries no team_id; APIStress just opens with the URL/method. The new run becomes a row in the user's current team. Safe. |
| **Existing data before the migration** | A "Legacy" team is auto-created. All pre-migration rows get its `team_id`. Admins decide what to keep. |
| **A user pastes the wrong key** | 401 with "Invalid access key" — no info about whether the key existed before. |
| **Two teams pick the same name** | Backend rejects with 409. |
| **No teams exist yet, first user pastes the env-var key** | Falls back to "Legacy" team for 30 days. After grace period: env-var key is rejected and we 404 the login until an admin creates a team. |

## 9. Migration plan

### Phase A — UI scaffold (✅ already shipped)
- `/admin` page with login + team CRUD using `localStorage`.
- Lets us iterate on UX before any DB change.

### Phase B — Backend tables + admin API · ~½ day
- Migration `003_teams.sql` with `teams`, `team_keys`, `team_members`.
- `/api/admin/*` routes wired to the same Gin router, gated by env-var
  `CH_ADMIN_KEY`.
- Frontend `/admin` page swaps `localStorage` for the live API.

### Phase C — Scope existing data · ~½ day
- `ALTER TABLE … ADD COLUMN team_id` on runs, tests, environments, pw_*
- One-time data backfill: legacy team gets all existing rows.
- Every handler gets the `team := TeamID(c)` line + `WHERE team_id = $team`.
- Login route returns `{team_id, team_name}`.
- UI shows the team chip.

### Phase D — Cleanup · ~few hours
- Disable the env-var fallback after 30 days.
- Add `last_used_at` updates on each key auth (so admins can see staleness).
- Add the audit log feed in admin UI.

**Total: ~1.5 days** of focused work to go from "prototype" to "running in
production with team isolation".

## 10. What we explicitly skip in v1

These are queued for the larger multi-tenant push (NEXTPLAN.md):

- Per-user identity (SSO / email login)
- Roles inside a team (member / lead)
- Cross-team sharing
- Per-user quotas
- Postgres Row-Level Security policies as a backstop
- Rate limiting per IP / per team

For 1,000 users in a trusted org behind a VPN, **team-level isolation is
already a huge step up** from where we are. The extra layers in NEXTPLAN.md
become essential when we open the tool to less-trusted users (contractors,
third parties).

## 11. Open questions for you to confirm before we wire Phase B

1. **Key format** — `<team-slug>-xxxx-yyyy-zzzz` (looks neat) or just
   `<random 32 chars>` (more secure-looking)?
2. **Admin passphrase rotation** — is once a quarter fine, or should we hash
   it and put it in a tiny `admins` table from day 1?
3. **Member visibility** — should the admin see *which user (display name)

   one-click expand.)
4. **Default team for the legacy CH_ACCESS_KEY** — name it `Legacy`, `Choice
   Techlab`, or something else?
5. **Quotas** — show them in the admin team card (read-only) or wait for
   NEXTPLAN.md Phase 4? (My default: wait.)
